# Revision Workflow

Define a revision process that is transparent, traceable, and protected from scope creep.

## What You Do

You help establish the operational backbone of design revision cycles — from how feedback is collected to how changes are tracked and signed off.

## Revision Workflow Stages

1. **Submission**: Designer shares deliverable with framing note — version, what changed, what feedback is needed
2. **Review window**: Client reviews during an agreed window (typically 3–5 business days)
3. **Feedback consolidation**: Feedback is collected in a single channel (not email + Slack + verbal) and consolidated by the account lead
4. **Triage**: Feedback is sorted into: implement, discuss, decline (out of scope), or defer
5. **Change request**: Out-of-scope items are documented and routed through the change request process with estimated time/cost impact
6. **Revision**: Designer implements agreed changes and updates version number
7. **Sign-off**: Client formally approves the revised deliverable — verbal approval is followed up in writing

## Version Control Convention

- Use semantic versioning for deliverables: `v1.0`, `v1.1` for minor revisions, `v2.0` for significant direction changes
- Never overwrite a previous version — keep the history
- File names include version and date: `project-name_homepage_v2.1_2025-06-10`

## Change Request Process

- Any work outside the agreed scope is a change request
- Change requests include: description of the change, estimated hours, estimated cost, and impact on timeline
- Change requests require written client approval before work begins

## Revision Round Limits

- Define the number of revision rounds included per deliverable in the contract
- Track rounds actively — notify the client proactively when the final included round is being used
- Additional rounds are billed at the agreed hourly or day rate
